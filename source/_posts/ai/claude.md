# 构建 Agent Harness 时，最难的是设计它的 Action Space

构建一个 agent harness，最难的问题之一，是如何设计它的 **action space（动作空间）**。

Claude 通过 **Tool Calling** 与外部环境交互。但在 Claude API 中，工具并不只有一种构建方式。你可以基于多种基础原语来组织工具，例如 `bash`、`skills`，以及近来加入的 **code execution（代码执行）**。

面对这些选项，一个关键问题就出现了：你应该如何为 agent 设计工具体系？

是只给它一个足够通用的工具，比如代码执行或 bash？还是反过来，为它可能遇到的每一种场景都单独设计一个工具，最后堆出几十个工具？

## 从模型视角理解工具设计

为了更好理解这个问题，可以尝试站在模型的角度思考。

假设你面前有一道很难的数学题。为了把它解出来，你最希望拥有哪些工具？答案显然取决于你本身具备什么能力。

纸和笔是最基础的工具，但你会受限于手工计算。  
计算器更进一步，但前提是你知道如何使用它的高级功能。  
而最快、最强的工具无疑是计算机，但前提是你知道如何借助它编写并执行代码。

这其实就是设计 agent 的一个有效框架：  
**你给它的工具，应该匹配它已经具备的能力边界。**

但新的问题是：你怎么知道模型到底擅长什么、不擅长什么？

答案并不复杂：持续观察它、阅读它的输出、反复实验。  
你需要学会“像 agent 一样思考”。

下面这些经验，就是在构建 Claude Code 的过程中，通过持续观察 Claude 得到的一些结论。

# 提升 Elicitation：AskUserQuestion 工具的设计演化

在设计 **AskUserQuestion** 工具时，我们的目标，是提升 Claude 的提问能力，也就是通常所说的 **elicitation**。

Claude 当然可以直接用自然语言向用户提问，但我们发现，这种方式的交互成本偏高。用户回答这些问题时，往往会感觉流程拖沓、效率不高。

于是问题变成了：  
**怎样降低交互摩擦，并提升用户与 Claude 之间的信息交换带宽？**

## 尝试一：修改 ExitPlanTool

我们做的第一个尝试，是给 `ExitPlanTool` 增加一个参数，让它在输出计划的同时，附带一个问题数组。

这是最容易实现的方案，但效果并不好。因为它把“输出计划”和“就计划提问”这两件事混在了一起，容易让 Claude 困惑。

例如：

- 如果用户的回答与原计划冲突怎么办？
- Claude 是否需要再次调用 `ExitPlanTool` 来修正计划？
- 一个工具里同时承载计划与追问，边界是否过于模糊？

很快我们意识到，这不是一个理想方案。

## 尝试二：修改输出格式

第二种方法，是修改 Claude 的输出格式要求，让它以一种可解析的 Markdown 结构来提问。

例如，可以要求它输出一组带选项的项目符号问题，然后再由前端把这段内容解析成 UI。

这个方案的优点是通用，且模型表面上也能较好地遵循这种格式。  
但问题在于：**不稳定。**

Claude 有时会：

- 额外补充无关句子
- 漏掉选项
- 临时改用别的格式

也就是说，这种方式“经常能用”，但“不可靠”。

## 尝试三：单独设计 AskUserQuestion 工具

最后，我们选择了第三种方案：  
直接设计一个独立的 **AskUserQuestion** 工具。

Claude 可以在任意时刻调用它，但在 **plan mode** 下会被特别鼓励这样做。  
当这个工具被触发时，系统会弹出一个模态框展示问题，并暂停 agent loop，直到用户完成回答。

这个方案带来了几个明显优势：

1. 可以强制 Claude 以结构化方式输出问题
2. 更容易保证问题里包含多个选项
3. 便于在 Agent SDK 或 skills 中复用
4. 工具边界清晰，不再和其他职责混淆

更关键的是：**Claude 本身似乎很愿意调用这个工具，而且调用效果不错。**

这一点其实非常重要。  
一个工具设计得再精妙，如果模型不知道该怎么调用，或者不愿意调用，它依然不会真正发挥作用。

这是否已经是 Claude Code 中 elicitation 的最终形态？未必。  
因为后面的例子会说明：**适用于一个模型的方法，未必就是另一个模型的最优解。**

# 随着模型能力变化，工具也要跟着变化：从 Todos 到 Tasks

在 Claude Code 刚上线时，我们很快意识到，模型需要一个 **Todo List** 来维持任务方向感。

于是我们提供了 `TodoWrite` 工具，让 Claude 可以：

- 创建 Todo
- 更新 Todo
- 在执行过程中逐项勾选
- 把 Todo 展示给用户

这在初期很有帮助，因为它让模型有了一个显式的任务跟踪机制。

但即便如此，我们仍然经常看到 Claude 忘记自己正在做什么。  
为了解决这个问题，我们一度在系统层每 5 个 turn 插入一次提醒，反复告诉它当前目标。

## 模型变强之后，原有工具可能反而变成束缚

随着模型能力提升，情况开始变化。

我们发现，模型不仅不再那么需要 Todo 提醒，反而会被它限制住。  
系统不断提醒 Todo List，会让 Claude 误以为自己必须严格遵守已有列表，而不是根据新情况主动调整任务。

与此同时，Opus 4.5 在使用 **subagents（子代理）** 上的表现也明显提升。  
这时，一个新问题出现了：

**如果多个子代理要协同工作，它们如何共享和协调一个固定的 Todo List？**

于是我们意识到，原来的 `TodoWrite` 已经不再适合新的能力形态。

## 从 TodoWrite 切换到 Task Tool

后续，我们用 **Task Tool** 替代了 `TodoWrite`。

这两者的核心差异在于：

- **Todos** 的目标，是让模型“不跑偏”
- **Tasks** 的目标，是让多个 agent “更好协同”

相比 Todo，Task 更适合多 agent 场景，因为它支持：

- 任务依赖关系
- 跨子代理共享更新
- 任务的修改与删除

这背后的经验很直接：

> 随着模型能力提升，过去“必需”的工具，今天可能已经在限制模型。

因此，工具设计不是一次性工作，而需要持续回看和重估。  
这也是为什么，在实际产品中，维持一组能力画像相对接近的模型，往往更容易做工具体系设计。

# 搜索接口设计：让模型自己构建上下文

对 Claude 来说，一类极其重要的工具，是那些能够帮助它 **主动构建上下文** 的搜索工具。

在 Claude Code 早期，我们使用 RAG 向量数据库为 Claude 提供上下文。

RAG 的优点很明显：

- 快
- 强大
- 检索能力好

但问题同样明显：

- 需要建立索引
- 需要额外配置
- 在不同环境中可能比较脆弱
- 更重要的是：上下文是“给”Claude 的，而不是“由 Claude 自己找到”的

## 如果它能搜网页，为什么不能搜代码库？

后来我们开始思考一个问题：

如果 Claude 能搜索网络，那为什么不能搜索代码库本身？

于是，我们给 Claude 增加了一个 **Grep 工具**。  
有了它，Claude 就可以自己检索文件、主动拼装上下文，而不是被动接收一段预先准备好的材料。

随着模型能力提升，我们越来越清楚地看到一个趋势：

> 只要提供合适工具，Claude 会越来越擅长自己构建所需上下文。

这也是后来 **Agent Skills** 所正式引入的理念：  
**Progressive Disclosure（渐进披露）**

---

# Progressive Disclosure：不用加新工具，也能扩展能力

所谓渐进披露，本质上是让 agent 通过探索逐步发现自己需要的信息，而不是在一开始就把所有信息都塞进上下文。

例如，Claude 可以先读取一个 skill 文件，而这个文件又可以引用其他文件，模型再递归读取这些内容。  
这样一来，它就能按需逐层展开信息，而不是一次性吞下全部背景。

实际上，skills 的一个常见用途，就是为 Claude 增加“搜索型能力”，例如：

- 告诉它某个 API 应该怎么调用
- 告诉它数据库该如何查询
- 告诉它某个系统接口的使用方式

在过去一年里，Claude 在这方面的能力变化非常明显：  
它从几乎无法主动建立上下文，逐步发展到可以跨越多层文件进行嵌套搜索，最终找到自己真正需要的那部分信息。

因此，**渐进披露** 现在已经成为一个非常常用的设计模式：  
**在不新增工具的前提下，给模型增加新的功能空间。**

# Claude Code Guide Agent：通过子代理扩展动作空间

目前 Claude Code 大约有 20 个工具，而我们始终在反复问自己：  
这些工具真的都必要吗？

因为每增加一个工具，模型就多了一个要考虑的选项。  
工具越多，不一定越强，也可能越乱。

一个典型案例是：我们发现 Claude 并不真正了解“如何使用 Claude Code 本身”。

例如，如果用户问它：

- 如何添加 MCP？
- slash command 是什么？
- 某个功能怎么配置？

它经常无法给出合格回答。

## 为什么不把这些信息直接塞进 system prompt？

一种简单方法，是把所有这类帮助信息直接放进 system prompt。  
但问题在于，这些信息大多数时候并不会被问到。

如果把它们长期塞在上下文里，会带来两个问题：

1. 增加 context rot
2. 干扰 Claude Code 的主任务 —— 写代码

## 用渐进披露替代工具扩张

于是我们先尝试了一种渐进披露方案：  
给 Claude 一个文档链接，让它在需要时自己加载文档并搜索答案。

这个方法可行，但也有一个缺点：  
Claude 往往会把大量搜索结果读进上下文，只为了找到一个本来可以直接返回的答案。

因此，我们进一步构建了 **Claude Code Guide subagent**。

当用户询问 Claude 关于“它自己”的问题时，系统会提示它调用这个子代理。  
这个子代理内部带有详细指令，专门告诉它：

- 如何高效搜索文档
- 应该返回什么信息
- 应该省略什么内容

虽然这个方案仍不完美，Claude 在回答“如何配置自己”这类问题时偶尔还是会困惑，但整体效果已经明显优于过去。

更重要的是，我们做到了一点：

> **在没有新增工具的前提下，扩展了 Claude 的 action space。**

# 工具设计是一门艺术，不是一套公式

如果你期待的是一套严格、固定、可直接套用的工具设计规则，那这并不是那样的指南。

为模型设计工具，既是一门科学，也是一门艺术。  
它高度依赖于三个变量：

- 你使用的模型
- agent 的目标
- 它所运行的环境

因此，没有一套对所有 agent 都成立的标准答案。

真正有效的方法通常是：

- 持续实验
- 仔细阅读模型输出
- 频繁调整工具设计
- 不断尝试新的组织方式

归根到底，你需要学会的，是：

**像 agent 一样去观察问题。**
