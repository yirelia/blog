# Harness 设计，正在成为 Agentic Coding 的关键变量
> **原文**：[harness-design-long-running-apps](https://www.anthropic.com/engineering/harness-design-long-running-apps)

在 Agentic Coding 的前沿实践里，真正决定效果上限的，往往不只是模型本身，而是 **Harness 设计**。原文作者围绕两个彼此关联的问题展开了一系列实验：一是如何让 Claude 产出更高质量的前端设计，二是如何让它在较长时间内持续自主完成完整的软件开发任务。相关工作最初建立在团队此前的前端设计 skill 和长时运行 coding agent harness 基础之上，虽然这些方法已经显著优于基线，但最终都遇到了新的瓶颈。

为了突破这个瓶颈，作者尝试把一种更通用的 AI 工程思路同时应用到两个不同场景中：一个是高度主观、依赖审美判断的前端设计，另一个是更强调正确性与可用性的自主编程。其核心灵感来自 GAN，也就是生成对抗网络。基于这一思路，作者设计了一个由 **生成者（generator）** 和 **评估者（evaluator）** 组成的多智能体结构。关键点不只是让模型去“生成”，而是先把原本模糊的主观评价标准，转写为可执行、可打分、可迭代的规则体系。

在此基础上，这套方法又被进一步迁移到长时自主编程场景中。作者延续了早期 harness 实践里的两个有效经验：第一，把复杂构建任务拆成可处理的小块；第二，使用结构化产物在不同 session 之间完成上下文交接。最终，形成了一套由 **规划者（planner）—生成者（generator）—评估者（evaluator）** 构成的三智能体架构，能够在持续数小时的自动化 coding session 中构建功能完整的全栈应用。

## 为什么“朴素实现”经常失效

作者指出，早期实验已经证明：对于长时运行的 agentic coding 来说，harness 设计本身会显著影响最终表现。此前常见做法是：先由一个 initializer agent 把产品需求拆成任务列表，再由 coding agent 逐个功能实现，并通过 artifacts 在 session 之间传递上下文。社区里也逐渐出现了类似共识，例如通过 hooks 或脚本让 agent 持续迭代。

但问题并没有被彻底解决。任务一旦变复杂，agent 仍然容易“跑偏”。作者将这类失败总结为两个典型模式。

第一个问题是：当上下文窗口不断累积后，模型会逐渐失去长期任务中的连贯性；有些模型还会出现一种“上下文焦虑”，也就是当它感知自己接近上下文极限时，会提前进入“收尾状态”，导致任务被过早结束。为此，作者强调 **context reset** 的价值：不是简单压缩历史，而是直接清空上下文，用一个新 agent 接手，同时通过结构化 handoff artifact 传递前一阶段状态与下一步任务。这样既能缓解连贯性衰减，也能减少“上下文焦虑”。

作者还专门区分了 **compaction** 和 **reset**。前者是对历史对话做压缩摘要，让同一个 agent 继续工作；后者则是彻底给 agent 一个“干净上下文”。compaction 虽然保留连续性，但并不能彻底消除上下文焦虑；reset 的代价则是 orchestration 更复杂，token 开销和延迟也会增加。

第二个问题是 **自我评估失真**。当 agent 评估自己刚刚完成的工作时，往往会倾向于给出过于正面的判断。这个问题在设计类任务里尤其明显，因为设计没有像软件测试那样明确的二值验证标准；“一个界面是否高级”本身就是主观判断，模型在这种场景下更容易宽松打分。即便是在有明确可验证结果的软件任务里，模型也常常因为判断失误而影响任务推进。因此，把“执行者”和“评判者”拆开，是一个非常有效的改进方向。虽然 evaluator 本质上仍然是 LLM，也会天然偏向宽容，但**训练一个外部、偏怀疑型的 evaluator，远比让 generator 对自己严格得多要现实得多**。

## 前端设计：把主观审美转成可评分问题

作者最早把这套机制用在前端设计任务上，因为这里的“自评失真”最容易被观察到。默认情况下，Claude 往往会生成技术上没问题、但视觉上比较保守、套路化明显的界面。

围绕前端设计，作者形成了两个关键判断。第一，审美虽然无法被彻底量化，但可以通过一套明确的设计原则和偏好规则，转化为可评分的问题。比如，“这个设计美不美”很难稳定回答，但“这个设计是否符合我们定义的好设计原则”就更容易成为一个清晰的评估项。第二，一旦把前端生成和前端评审拆成两个 agent，就能形成一个持续驱动质量提升的反馈闭环。

基于此，作者为 generator 和 evaluator 都定义了四项评分标准：

### 1. 设计质量（Design Quality）
界面是否呈现为一个统一、连贯的整体，而不是零散元素的堆砌。优秀设计应当让配色、排版、布局、图像与细节共同塑造出鲜明的气质和身份感。

### 2. 原创性（Originality）
界面中是否存在真实的定制化决策，而不是模板布局、组件库默认样式或典型 AI 生成套路。一个人类设计师应当能识别出其中有意识的创意选择；未经修改的 stock 组件，或者“白底卡片+紫色渐变”这类典型 AI 痕迹，都应被判为不合格。

### 3. 工艺质量（Craft）
主要看技术层面的完成度，例如字重层级、间距一致性、色彩和谐、对比度等。这更像是“基本功检查”，而不是创意判断。多数合理实现默认就能过关，失败通常说明基础设计能力有问题。

### 4. 功能性（Functionality）
即便不考虑美观，用户是否也能清楚理解界面用途，快速找到主要操作，并顺利完成任务。

在权重上，作者明确强调 **设计质量和原创性高于工艺与功能性**。原因是 Claude 在 craft 和 functionality 上本来就表现不错，真正拉不开差距的，是设计是否有气质、是否摆脱“AI 味”。因此，这套标准会对高度泛化、模板化的“AI slop”进行明确惩罚，从而迫使模型在审美层面承担更大的探索风险。

为了让 evaluator 的判断更稳定，作者还使用了带有详细评分拆解的 few-shot 示例进行校准，尽量让评估器的审美口径与自己保持一致，并减少多轮迭代中的评分漂移。

在工程实现上，这套循环构建于 Claude Agent SDK 之上。流程是：先由 generator 生成 HTML/CSS/JS 前端，再由接入了 Playwright MCP 的 evaluator 直接操作正在运行的页面，自己浏览、截图、观察，再按照四项标准给出评分和详细批评。随后，这些反馈被回传给 generator，作为下一轮迭代的输入。整个过程通常会持续 5 到 15 轮，有些任务甚至要跑上 4 个小时。作者还要求 generator 在每轮结束后做一个策略判断：如果评分趋势向好，就继续在当前方向精修；如果方向本身不成立，就主动做大的风格转向。

从结果来看，evaluator 的评分通常会随着迭代上升，之后进入平台期，但仍留有提升空间。有些案例是渐进优化，有些则会在中途突然发生显著的审美跃迁。更有意思的是，作者发现 **评分标准的措辞本身，也在反向塑造 generator 的输出风格**。例如，当规则里写入“最好的设计应该有博物馆级别的质感”这类描述时，模型会朝某种特定视觉趋向收敛。这说明，prompt 中对标准的表述方式，不只是评估语言，也会变成生成方向的隐性约束。

作者还提到，虽然整体分数通常会逐轮提高，但提升曲线并不总是线性的。后期版本整体上经常更成熟，但也存在中间某一轮反而比最终轮更讨喜的情况。同时，随着轮次增加，生成方案的实现复杂度也往往上升，因为 generator 会根据 evaluator 的反馈主动尝试更激进、更复杂的设计解法。甚至在第一轮里，仅仅因为有了这些评分标准和相关表述，输出质量就已经明显高于完全不加约束的 baseline。

一个典型案例是：作者让模型生成一个“荷兰艺术博物馆”的网站。到第九轮时，它已经给出一个质量不错的深色落地页；但到第十轮，模型完全推翻前面的方案，改成一个具有空间感的 3D 展厅体验：用 CSS 透视渲染棋盘格地面，墙面上自由挂置艺术作品，页面导航也不再依赖滚动或普通点击，而是通过“穿门进入下一个展厅”的方式完成。这种跃迁式创意，并不是单次生成里常见的结果。

## 从前端实验走向全栈自主开发

在前端实验验证有效后，作者把这套带有 GAN 思路的 generator-evaluator 模式，进一步迁移到了全栈开发中。在软件工程语境下，这个结构其实非常自然：design evaluator 的位置，几乎就对应了现实开发流程里的 code review 和 QA。

这一次，作者没有停留在双 agent，而是构建了一个更完整的三智能体体系：

### Planner
Planner 的职责，是把用户给出的 1 到 4 句话短提示，扩展成一份完整产品规格说明。作者要求它在范围上尽可能大胆，但重点放在产品上下文和高层技术设计，而非过早给出过细的实现细节。因为如果 planner 在前期就写死大量技术实现，而其中某些判断又是错的，这些错误会沿着规格一路传递到后续实现中。相比之下，更合理的做法是：先约束交付目标，再让后续 agent 在实现过程中自己探索路径。作者还要求 planner 主动在产品方案中寻找可融合的 AI 特性。

### Generator
Generator 延续了之前“按功能逐项实现”的工作方式，但这次被组织成一个个 sprint。每个 sprint 从 spec 中选取一项功能目标，使用 React、Vite、FastAPI 和 SQLite（后期改为 PostgreSQL）去完成实现。作者还要求 generator 在每轮 sprint 结束时做自我检查，再把结果交给 QA；同时，它也具备 git 版本管理能力。

### Evaluator
Evaluator 则扮演真正的产品验收者。早期 harness 产出的应用常常“看起来不错”，但一旦上手点几下就会暴露很多真实 bug。为此，evaluator 通过 Playwright MCP 像真实用户一样点击运行中的应用，检查 UI 功能、API 接口和数据库状态，并基于“产品深度、功能完整性、视觉设计、代码质量”等标准对 sprint 进行打分。每个维度都有硬阈值，只要有一项不达标，这个 sprint 就会被判失败，generator 需要根据详细反馈继续修正。

另一个关键机制，是在每个 sprint 开始前，generator 和 evaluator 先协商一个 **sprint contract**，也就是先定义清楚这一小段工作“完成”的标准。之所以要多这一步，是因为 planner 产出的产品 spec 本身故意保持高层抽象，需要有一个中间桥梁，把用户故事转化为可测试、可验收的具体实现目标。generator 先提方案，说明本轮做什么、如何验证成功；evaluator 再检查这份方案是否真的对齐目标。双方不断调整，直到达成一致。

在 agent 间通信上，这套系统采用 **文件作为交接媒介**：一个 agent 写文件，另一个 agent 读取文件后再回复到同一文件或新文件中。随后 generator 再基于双方确认过的 contract 去编码实现，最后交给 evaluator 做 QA。这样的好处是，系统既能保持对 spec 的忠实，又不会在任务一开始就把实现路径限制得过死。

## 小结

这篇文章真正有价值的地方，不是简单证明“多智能体比单智能体更强”，而是揭示了一个更重要的结论：在 Agent 时代，**系统设计本身就是能力的一部分**。模型能力决定下限，但 harness 设计、任务拆解、上下文管理、评估闭环与 agent 协作方式，才决定了最终能否逼近上限。

换句话说，未来真正拉开差距的，未必只是“谁的模型更大”，而更可能是谁更懂得设计一套能让模型持续发挥、持续纠错、持续演进的工作系统。
